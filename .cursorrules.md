### 🐍 Python & Async Best Practices

- **Use Python 3.10+** with type hints throughout
- **All agent/tool methods must be async** — WitsV3 is fully async
- **Follow PEP8** and format with black
- **Use Pydantic models** for data validation (`core/schemas.py`)
- **Import style:** relative within packages, absolute cross-package

### 🧱 Code Structure & Modularity

- **Max 500 lines per file** — split when approaching the limit
- Maintain:
  - `agents/` — BaseAgent, orchestrators, control center, specialists
  - `core/` — config, LLM, memory, schemas, safe_code_editor
  - `tools/` — BaseTool implementations (auto-discovered)
  - `web/` — FastAPI + SSE UI
- New tools extend `BaseTool`; new agents extend `BaseAgent`

### 🧪 Testing Requirements

- Add pytest coverage under `/tests` mirroring package layout
- Use pytest-asyncio for async tests; mock Ollama and external I/O
- Per feature: happy path, error path, edge case

### 📚 WitsV3-Specific Conventions

- **StreamData** — `stream_thinking()`, `stream_action()`, etc.
- **Tool Registry** — register with `get_llm_description()`
- **Memory Manager** — `store_memory()` / `search_memory()`
- **Config-driven** — `WitsV3Config` / `config.yaml`; secrets in `.env`
- **Logging** — `self.logger` in classes

### 🧠 LLM Integration Rules

- **Ollama is primary** — prefer friendly errors via `core/user_errors.py` when down
- Stream via AsyncGenerator; parse robustly (`ResponseParser`, JSON repair paths)
- Prefer `core/model_router.py` for size routing over the dormant adaptive-LLM stack

### 📝 Documentation Standards

- Update **README.md** when install or user-facing behavior changes
- Google-style docstrings; `# Reason:` for non-obvious logic
- Prefer updating the canonical roadmap over proliferating “status” docs

### ⚠️ Critical WitsV3 Rules

- Never break async patterns — I/O must be async
- Preserve the ReAct loop in orchestrators
- Maintain compatibility with existing tools/agents unless intentionally retiring them
- Prefer Ollama running for integration checks
- Careful Unicode / memory-segment serialization

### 🚫 Don't Do

- Synchronous I/O in hot paths
- Circular imports between packages
- Schema changes without updating dependents
- Hardcoded config (use `config.yaml` / env)
- Leaving broken files on disk after a failed self-edit (use the verified-edit pipeline)
